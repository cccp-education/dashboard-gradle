== Réalisations

=== DDD — Modèle de domaine timeline

- `Milestone` : value object immuable (id, label, date, borough, dagLevel, type, status).
- `MilestoneType` : `EPIC_RELEASE`, `SESSION`, `SPRINT_GATE`.
- `MilestoneStatus` : `DONE`, `IN_PROGRESS`, `PLANNED`.
- `Timeline` : agrégat qui maintient les jalons triés par date puis par id, expose `milestonesUpTo(date)` et `byType(type)`.
- `TimelineBuilder` : domain service transformant `DashboardData` en `Timeline`.

Règle métier clé : un jalon EPIC est daté par la session de livraison extraite de `SESSIONS_HISTORY.adoc`. Si aucune session ne correspond, une date future par défaut est utilisée et le statut est `PLANNED`.

=== TDD — Tests unitaires

- `MilestoneTest` : 3 tests (propriétés, tri par date, tri par id si dates identiques).
- `TimelineTest` : 3 tests (tri, filtre par date, filtre par type).
- `TimelineBuilderTest` : 5 tests (mapping EPIC terminé → milestone daté, fallback planned, in-progress, skip sans borough, dag level).

=== Intégration dans le pipeline existant

- `EpicData` enrichi d’un champ `session` pour conserver le numéro de session de livraison.
- `IndexCrawler` parse le numéro de session dans la colonne statut (`✅ S000` → `000`).
- `DashboardPage` expose `timeline: Timeline`.
- `DashboardRenderer` construit la timeline via `TimelineBuilder` et la passe au template.
- `dashboard.thyme` : nouvelle section "Timeline" avec liste verticale, marqueurs colorés par statut.
- `styles.css` : styles `.timeline*`, marqueurs `.status-DONE`, `.status-IN_PROGRESS`, `.status-PLANNED`.

=== BDD — Cucumber

- Scénario `Generate dashboard site with timeline` dans `dashboard_generate.feature`.
- Step `givenFoundryWithCompletedEpics` dans `CrawlSteps.kt` pour créer un borough avec EPIC terminé et SESSIONS_HISTORY.
